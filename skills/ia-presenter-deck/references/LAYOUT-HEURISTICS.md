# iA Presenter Layout Heuristics

These rules exist because syntax correctness is not enough. A deck can be valid and still feel weak.
You are designing the visible layer, not only the notes.
Slides should support the presenter, not become projected documents.

## Use Side-by-Side Sparingly

Good reasons:
- contrast
- before / after
- customer / team
- weak / strong example
- image panel plus argument panel

Bad reasons:
- to spread out two unrelated short phrases
- to make a sparse slide look more designed

## Prefer Stacked When the Argument Builds

If one point explains, sharpens, or complicates the one before it, stacked is usually better than split.

## Use Headings as Layout, Not Decoration

`#`, `##`, `###`, and `####` are part of the visual system.

Use them to:
- reset a section
- stack a claim and qualifier
- create pace without another list
- land a line that should be seen before it is explained

## Do Not Strand Tiny Content

If one side of a split has only one short sentence, the slide is probably under-designed or over-split.

Fix by:
- merging into one stacked slide
- turning it into a stronger comparison
- promoting one line to a bigger closing-style slide
- turning the split into a deliberate image-panel slide with enough text weight on the other side

## Tables Need a Reason

Use a table only when:
- the audience needs exact deltas
- three rows are better than three slides
- the grid is the message

When the table is the message:
- keep the heading and table in the same intentional stack if you want a full-width proof slide
- avoid letting the heading become a stranded left column unless comparison is the actual point
- consider a wider 3-column proof table when the slide needs to occupy the full frame cleanly

## Cells Decide Composition

iA Presenter layouts respond to cells, not just content types.

Default:
- one element per cell
- title plus subtitle can share a cell
- title stack plus table can share a cell when you want a top-to-bottom proof slide

Use a blank line when you want a true split.
Remove the blank line when the slide wants a stacked composition instead of side-by-side.

## Use the Layout Picker On Purpose

The auto-layout is a starting point.

Always review slides that contain:
- a table
- an image plus text
- three or more visible elements
- a section title that feels stranded away from the proof it introduces

Otherwise, use a line, comparison, or distilled proof statement.

## Closers Need Weight

A proper final slide should feel like a landing, not a polite stop.

Better closers:
- Start with one queue.
- Smaller quarter. Stronger business.
- Fewer tools. Faster confidence.

Weaker closers:
- Thank you
- Questions
- Next steps

## Notes Should Carry the Temperature

If the slide is short and the deck feels cold, the notes are probably too flat.

Notes should:
- add voice
- add movement
- set up the next slide
- turn a label into an argument

Notes should not:
- contain the only important claim on the slide
- carry the whole proof while the visible surface stays empty
- rescue a slide that has no visible payload

## Visible Rhythm Matters

If every slide has the same silhouette, the deck will feel flatter than it needs to.

Vary with:
- heading stacks
- chapter cards or section resets
- ordered sequences
- quote-led slides
- comparison slides
- tiny proof tables
- one-line landings

## Lists Need Compression

Lists are useful, but they become wallpaper fast.

Prefer:
- 3-4 items over 6-8
- stronger verbs over abstract nouns
- one list followed by a different surface on the next slide

If the deck already has two list-heavy slides in a row, the next slide should probably not be another list.

## Simplicity Is Not Emptiness

Good simplicity:
- one strong thought
- one clear proof object
- one image with a reason
- one composition that reads quickly

Bad simplicity:
- a nearly empty slide that communicates nothing
- a generic stock image filling silence
- a tiny proof object stranded in whitespace
- visual restraint used as an excuse not to decide

## Candidate Mode Should Vary Rhythm

When generating multiple directions, vary:
- headline style
- slide count and pacing
- how early the decision appears
- whether the deck is more contrast-led, proof-led, or story-led
